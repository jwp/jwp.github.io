DRY Constraints
2023-04-22

I often find identifier re-use implies a consistent type. While readability may suffer without redundancy, it may be advantageous to be able to constrain idenitifiers within some scope(source file, directory, project). In particular, identifiers that closely resemble the type's name.

The goal being DRY constraints, simplified element directory associations(for completion indexes), and simplified refactoring tools? Not such a feature would fit into many languages gracefully. Exclusive words? Nearly a keyword treated as an identifier.
